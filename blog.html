Building High-Performance REST APIs in SAP ECC with ICF Handlers

In modern enterprise architectures, REST APIs have become the de facto standard for system integration. However, building scalable APIs in SAP ECC that can handle millions of records while maintaining performance requires careful architectural decisions and optimization techniques.

The Challenge: 140 Million Records

Recently, I worked on a project where we needed to expose pricing data from SAP ECC to a Liferay portal. The challenge? We had 140 million records spread across 9 brands, with weekly data volatility. The existing implementation made 18 API calls per sync, resulting in frequent timeouts and poor user experience.

"The key to scaling isn't just about handling more data—it's about designing systems that remain efficient as data grows exponentially." - Architecture Principle

Solution Architecture

We redesigned the API using a three-tier approach:

  • Pre-calculated Snapshot Table: Weekly batch job stores pricing data in ZPRICING_SNAPSHOT
  • Cursor-based Pagination: Stable cursors enable resumable data transfer
  • ICF Handler Implementation: Custom REST endpoint with IF_HTTP_EXTENSION interface

1. Database Design

The foundation of any high-performance API is proper database design. Here's our Z-table structure: 

@EndUserText.label : 'Pricing Snapshot for REST API'
@AbapCatalog.tableCategory : #TRANSPARENT

DEFINE TABLE ZPRICING_SNAPSHOT {
  KEY CLIENT    : MANDT;
  KEY BRAND     : CHAR3;
  KEY MATNR     : MATNR;
  KEY KUNNR     : KUNNR;
  
  MAKTX         : MAKTX;
  KBETR         : KBETR_KOND;
  WAERS         : WAERS;
  CHANGE_TSTMP  : TIMESTAMP;
  RECORD_STATUS : CHAR1;
}

Notice the composite key (CLIENT, BRAND, MATNR, KUNNR) which aligns with our cursor pagination strategy. The CHANGE_TSTMP field is crucial for delta sync capabilities.

2. Critical Indexes

Without proper indexes, even the best API design will fail. We created three strategic indexes: 

* Index 1: Delta Query Optimization
INDEX ZPS_IDX1 ON ZPRICING_SNAPSHOT
  FIELDS: BRAND CHANGE_TSTMP MATNR KUNNR.

* Index 2: Pagination Cursor Optimization  
INDEX ZPS_IDX2 ON ZPRICING_SNAPSHOT
  FIELDS: BRAND MATNR KUNNR CHANGE_TSTMP.

Index ZPS_IDX2 is the hero here—it enables sequential reads for pagination, reducing random I/O by 95%.

3. ICF Handler Implementation

The REST API handler implements cursor-based pagination using a simple but powerful pattern: 

METHOD fetch_pricing_data.
  DATA: lv_cursor_brand TYPE mvgr1,
        lv_cursor_matnr TYPE matnr,
        lv_cursor_kunnr TYPE kunnr.

  * Decode cursor if provided
  IF iv_cursor IS NOT INITIAL.
    decode_cursor(
      EXPORTING iv_cursor = iv_cursor
      IMPORTING ev_brand  = lv_cursor_brand
                ev_matnr  = lv_cursor_matnr
                ev_kunnr  = lv_cursor_kunnr ).
  ENDIF.

  * Fetch with cursor-based WHERE clause
  SELECT *
    FROM zpricing_snapshot
    WHERE brand = @iv_brand
      AND record_status = 'A'
      AND ( brand > @lv_cursor_brand OR
            ( brand = @lv_cursor_brand AND matnr > @lv_cursor_matnr ) OR
            ( brand = @lv_cursor_brand AND matnr = @lv_cursor_matnr 
              AND kunnr > @lv_cursor_kunnr ) )
    ORDER BY brand, matnr, kunnr
    INTO TABLE @et_data
    UP TO @iv_limit ROWS.

  * Generate next cursor
  IF lines( et_data ) = iv_limit.
    READ TABLE et_data INDEX iv_limit INTO DATA(ls_last).
    ev_next_cursor = encode_cursor(
      iv_brand = ls_last-brand
      iv_matnr = ls_last-matnr
      iv_kunnr = ls_last-kunnr ).
  ENDIF.
ENDMETHOD.

Performance Results

The optimized API delivered impressive results:

  • 150,000 records per page (4.6 MB compressed with GZIP)
  • 933 total pages for 140M records
  • 31 minutes for complete sync (vs. previous timeout issues)
  • Memory usage: ~300 MB per request (well within SAP work process limits)

Key Takeaways

  1. Pre-calculation wins: Weekly snapshots eliminate real-time calculation overhead
  2. Indexes are mandatory: Without proper indexes, 140M records = disaster
  3. Cursor pagination scales: Unlike offset-based pagination, cursors maintain O(1) performance
  4. GZIP compression matters: 87% size reduction (35 MB → 4.6 MB) drastically improves network transfer
  5. Plan for parallelization: After HANA migration, parallel requests can reduce sync time to ~8 minutes

Conclusion

Building high-performance REST APIs in SAP ECC requires a holistic approach—from database design and indexing to ABAP implementation and network optimization. The cursor-based pagination pattern proved crucial for handling enterprise-scale data volumes while maintaining sub-second response times.
